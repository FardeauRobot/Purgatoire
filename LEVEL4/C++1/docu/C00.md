#### The Namespace Problem

In C, every function in the standard library lives in the global scope.
If you write a function called `printf`, you just shadowed the real one — and the compiler won't warn you.

C++ fixes this with **namespaces**: a named container that scopes symbols away from each other.

```
Global scope in C:          Global scope in C++:
┌───────────────────┐       ┌──────────────────────────────────────┐
│ printf            │       │  std::                               │
│ strlen            │       │  ┌──────────────────────────────┐    │
│ toupper           │       │  │  cout   cin   string  endl   │    │
│ malloc            │       │  │  vector map   sort    find   │    │
│ YOUR functions    │       │  └──────────────────────────────┘    │
│  → name collisions│       │  YOUR functions (safe, separate)     │
└───────────────────┘       └──────────────────────────────────────┘
```

The `std::` prefix is just you saying: *"I want the `cout` that lives inside the `std` namespace"*.

---

#### C vs C++ — I/O

```c
/* C */
#include <stdio.h>
#include <ctype.h>

int main(int argc, char **argv) {
    for (int i = 0; argv[1][i]; i++)
        printf("%c", toupper(argv[1][i]));
    printf("\n");
}
```

```cpp
// C++
#include <iostream>
#include <cctype>

int main(int argc, char **argv) {
    for (int i = 0; argv[1][i]; i++)
        std::cout << (char)std::toupper(argv[1][i]);
    std::cout << std::endl;
}
```

Key differences:
| C | C++ |
|---|-----|
| `#include <stdio.h>` | `#include <iostream>` |
| `printf("%c", x)` | `std::cout << x` |
| `\n` in format string | `std::endl` (also flushes buffer) |
| `toupper` global | `std::toupper` namespaced |

> **Mental model:** `std::cout <<` is a pipeline. You push data into it with `<<` like water flowing left to right. `std::endl` closes the pipe and flushes it.

#### From C `struct` to C++ `class`

In C, a `struct` is just a bag of data. You pass it around and anyone can read or write to any field:

```c
/* C — no protection, no methods */
typedef struct {
    char first_name[50];
    char phone[20];
    int  index;
} Contact;

void fill_contact(Contact *c, const char *name) {
    strcpy(c->first_name, name); // anyone can do this from anywhere
}
```

In C++, a `class` bundles the data **and** the functions that operate on it, and controls who can access what:

```cpp
// C++ — data is protected, access is controlled
class Contact {
private:
    std::string m_first_name;  // only Contact's own methods can touch this
    std::string m_phone;

public:
    void setName(std::string name) { m_first_name = name; } // the only door in
    std::string getName() const    { return m_first_name; } // the only door out
};
```

```
C struct:                        C++ class:
┌──────────────────┐             ┌─────────────────────────────────┐
│ first_name       │◄── anyone   │  PRIVATE                        │
│ phone            │◄── can      │  ┌───────────────────────────┐  │
│ index            │◄── touch    │  │ m_first_name              │  │
└──────────────────┘             │  │ m_phone                   │  │
                                 │  │ m_index                   │  │
                                 │  └───────────────────────────┘  │
                                 │           ▲  only accessible     │
                                 │           │  through:            │
                                 │  PUBLIC   │                      │
                                 │  ┌────────┴──────────────────┐  │
                                 │  │ setName()   getName()      │  │
                                 │  │ setPhone()  getPhone()     │  │
                                 │  └───────────────────────────┘  │
                                 └─────────────────────────────────┘
```

> **Why does this matter?** When your program has a bug where a contact's name gets corrupted, in C you have no idea where it happened — anyone could have written to it. In C++, you know it happened inside `setName()`. The blast radius of a bug shrinks dramatically.

#### The Orthodox Canonical Form (OCF)

Every class in C++ must define these 4 members. If you don't, the compiler generates them for you — and the auto-generated versions do **shallow copies**, which breaks anything that owns heap memory.

```
                    ┌──────────────────────────────────────────┐
                    │           Orthodox Canonical Form         │
                    │                                          │
  Contact c1;       │  1. Default Constructor                  │
  ────────────────► │     Contact();                           │
                    │     → called when you create an empty    │
                    │       object                             │
                    │                                          │
  Contact c2(c1);   │  2. Copy Constructor                     │
  ────────────────► │     Contact(const Contact& other);       │
                    │     → called when creating FROM another  │
                    │                                          │
  c2 = c1;          │  3. Copy Assignment Operator             │
  ────────────────► │     Contact& operator=(const Contact&);  │
                    │     → called when ASSIGNING to existing  │
                    │                                          │
  }  ← scope ends   │  4. Destructor                           │
  ────────────────► │     ~Contact();                          │
                    │     → called when object goes out of     │
                    │       scope or is deleted                │
                    └──────────────────────────────────────────┘
```

**Copy constructor vs assignment operator — the difference:**

```cpp
Contact c1;          // default constructor
Contact c2(c1);      // copy constructor    — c2 doesn't exist yet, being BORN from c1
Contact c3 = c1;     // copy constructor    — same thing, c3 is being born
c2 = c1;             // assignment operator — c2 already EXISTS, being OVERWRITTEN
```

A correct copy constructor in C++:
```cpp
// WRONG (your current code — two bugs)
Contact::Contact(const Contact& other)
    : m_first_name(other.m_first_name),
      m_last_name(other.m_last_name),
      m_nickname(other.m_last_name),      // copies last_name instead of nickname!
      m_darkest_secret(other.m_darkest_secret),
      m_index(other.m_index) {}           // m_phone_number is missing!

// CORRECT
Contact::Contact(const Contact& other)
    : m_first_name(other.m_first_name),
      m_last_name(other.m_last_name),
      m_nickname(other.m_nickname),       // fixed
      m_phone_number(other.m_phone_number), // added
      m_darkest_secret(other.m_darkest_secret),
      m_index(other.m_index) {}
```

#### C vs C++ — Strings

```c
/* C — manual memory, manual length, error-prone */
char name[50];
strcpy(name, "Alice");
int len = strlen(name);
char *sub = malloc(4);
strncpy(sub, name, 3);
sub[3] = '\0';
// don't forget to free(sub)...
```

```cpp
// C++ — std::string handles everything
std::string name = "Alice";
int len = name.length();         // 5
std::string sub = name.substr(0, 3); // "Ali" — no malloc, no free
bool empty = name.empty();       // false
```

```
C string:                        C++ std::string:
┌─────────────────────────┐      ┌────────────────────────────────┐
│ char name[50]           │      │ std::string name               │
│ A l i c e \0 _ _ _ ...  │      │  ┌──────────────────────────┐  │
│ ▲ you manage the size   │      │  │ data: A l i c e \0       │  │
│ ▲ you track the length  │      │  │ length: 5                │  │
│ ▲ you handle overflow   │      │  │ capacity: 16 (managed)   │  │
└─────────────────────────┘      │  └──────────────────────────┘  │
                                 │  → resizes automatically        │
                                 │  → knows its own length         │
                                 │  → no buffer overflow           │
                                 └────────────────────────────────┘
```

#### Header Guards — Why They Exist

Imagine you include the same header twice (which happens easily through transitive includes):

```
main.cpp includes:
  ├── PhoneBook.hpp  (includes Contact.hpp)
  └── Contact.hpp   (included again — DOUBLE DEFINITION → compiler error)
```

Header guards prevent the second include from doing anything:

```cpp
#ifndef CONTACT_HPP   // "if CONTACT_HPP is not yet defined..."
#define CONTACT_HPP   // "...define it now and process what follows"

class Contact { ... };

#endif                // "end of the guarded block"
```

On the second inclusion, `CONTACT_HPP` is already defined, so everything between `#ifndef` and `#endif` is skipped.

`#pragma once` does the same thing in one line and is accepted at 42:
```cpp
#pragma once

class Contact { ... };
```

#### The 3 Meanings of `static`

`static` is one of the most overloaded keywords in C/C++. Context is everything:

```
1. Inside a class → ONE shared copy for all instances
   ┌──────────────────────────────────────────────────┐
   │  class Account {                                 │
   │      static int _nbAccounts;  // shared by all   │
   │  };                                              │
   └──────────────────────────────────────────────────┘

2. Inside a function → survives between calls
   ┌──────────────────────────────────────────────────┐
   │  void count() {                                  │
   │      static int n = 0; // initialized once       │
   │      n++;              // persists next call      │
   │  }                                               │
   └──────────────────────────────────────────────────┘

3. At file scope (outside any class/function) → invisible to other files
   ┌──────────────────────────────────────────────────┐
   │  // helper.cpp                                   │
   │  static int secret = 42; // only THIS file sees  │
   └──────────────────────────────────────────────────┘
```

---

#### Static Members — Memory Layout

```
Without static — each object has its own copy:

Account a1(42):  ┌──────────────────┐
                 │ _accountIndex: 0 │
                 │ _amount: 42      │
                 │ _nbDeposits: 0   │
                 │ _nbAccounts: 0   │ ← its own copy, useless
                 └──────────────────┘

Account a2(54):  ┌──────────────────┐
                 │ _accountIndex: 1 │
                 │ _amount: 54      │
                 │ _nbDeposits: 0   │
                 │ _nbAccounts: 0   │ ← another copy, no link to a1's
                 └──────────────────┘


With static — one shared copy, owned by the CLASS:

                 Account::_nbAccounts: 2   ← lives here, shared
                 Account::_totalAmount: 96 ← lives here, shared
                         ▲                         ▲
                         │                         │
Account a1:  ┌───────────┴──────┐    Account a2:  ┌┴─────────────────┐
             │ _accountIndex: 0 │                  │ _accountIndex: 1 │
             │ _amount: 42      │                  │ _amount: 54      │
             │ _nbDeposits: 0   │                  │ _nbDeposits: 0   │
             └──────────────────┘                  └──────────────────┘
```

#### Static Member Functions — Why They Can't Access Instance Data

```cpp
// This fails — and here's exactly why:
void Account::displayAccountsInfos(void) {  // static function
    std::cout << _amount;  // ERROR
}
```

```
When you call a regular method:          When you call a static method:

Account a1(42);                          Account::displayAccountsInfos();
a1.displayStatus();                               ▲
      ▲                                           │
      │ compiler secretly passes:                 │ NO object passed.
      │   this = &a1                              │ No "this".
      ▼                                           │ Which _amount?
void displayStatus() {                            │ a1's? a2's? doesn't exist.
    std::cout << this->_amount; // = 42 ✓        ▼
}                                        void displayAccountsInfos() {
                                             std::cout << _amount; // ??? ✗
                                         }
```

> The rule is simple: if a function has no reason to know *which* object it's called on, make it `static`. Global stats (`_nbAccounts`, `_totalAmount`) don't belong to any one object — so the function that displays them shouldn't either.

#### Declaration vs Definition — The Linker Gap

This is one of the most confusing things for C++ beginners. Here's the full picture:

```
Step 1 — Compilation (per .cpp file):
  Account.cpp ──► compiler ──► Account.o  (object file)
  tests.cpp   ──► compiler ──► tests.o    (object file)

  The compiler only reads ONE .cpp at a time.
  It trusts the .hpp declarations: "ok, _nbAccounts exists somewhere"

Step 2 — Linking (combines all .o files):
  Account.o + tests.o ──► linker ──► executable

  The linker looks for the ACTUAL memory for _nbAccounts.
  If you never wrote:
      int Account::_nbAccounts = 0;  ← in a .cpp
  The linker finds nothing → "undefined reference" error
```

```cpp
// Account.hpp — DECLARATION (no memory allocated, just a promise)
class Account {
    static int _nbAccounts;   // "this will exist somewhere"
};

// Account.cpp — DEFINITION (memory is actually reserved here)
int Account::_nbAccounts = 0;  // "here it is, starts at 0"
//  ^^^^^^^^^^^^ — the :: ties it back to the class
//  no `static` keyword here — that only goes in the declaration
```

#### `const` Member Functions

```cpp
int checkAmount() const;
//                ^^^^^ this means: "I promise not to modify the object"
```

```
Without const:                     With const:
a.checkAmount()                    a.checkAmount()
      │                                  │
      ▼                                  ▼
int checkAmount() {                int checkAmount() const {
    _amount = 0; // allowed ✓         _amount = 0; // COMPILER ERROR ✗
    return _amount;                    return _amount;        // fine ✓
}                                  }
```

**The practical benefit — it enables read-only references:**
```cpp
void printInfo(const Account& a) {   // you promised not to modify a
    a.checkAmount();  // only works if checkAmount() is const
    a.makeDeposit(5); // compiler error — makeDeposit is not const
}
```

Use `const` on every method that only reads. It's not optional — it's part of designing a correct interface.

#### C vs C++ — Time Formatting

```c
/* C */
#include <time.h>
time_t now = time(NULL);
struct tm *t = localtime(&now);
char buf[16];
strftime(buf, sizeof(buf), "%Y%m%d_%H%M%S", t);
printf("[%s] ", buf);
```

```cpp
// C++ (C++98 — same functions, different namespace)
#include <ctime>
std::time_t now = std::time(NULL);
std::tm*    t   = std::localtime(&now);
char        buf[16];
std::strftime(buf, sizeof(buf), "%Y%m%d_%H%M%S", t);
std::cout << "[" << buf << "] ";
```

The C++ versions are the same functions — just accessed through `std::`. The `<ctime>` header is the C++ wrapper around C's `<time.h>`.

## Part 3 — Notions We Worked Through Together

These are the things that weren't obvious from reading — concepts that need to be *explained*, not just shown:

- **Why `static` isn't written in the `.cpp` definition** — the `static` keyword in a class context is a declaration-only qualifier. Putting it in the `.cpp` definition would mean something else entirely (file-scope static), so the compiler rejects it.

- **Why a `static` member function crashes when you try to use `_amount`** — no `this` pointer exists, so instance variables have no address to resolve to.

- **When "created" prints** — the vector constructor on line 24 of `tests.cpp` calls `Account(int)` eight times immediately. The print is inside the constructor, so it fires during that line, before `main()` does anything else visible.

- **The linker error vs the compiler error** — the compiler compiles one file at a time and trusts declarations. The linker assembles everything and demands real definitions. A missing static variable definition compiles fine but fails at link time.

- **`std::for_each` + `std::mem_fun_ref`** — this is C++98's verbose way of saying "call this method on every element." It's equivalent to a range-for loop with a method call, just written inside-out.

- **Static member initialization timing** — happens before `main()`, in the startup phase. This is why global state through static members is predictable: by the time your constructor runs, `_nbAccounts` already exists and is already `0`.

## Part 4 — Tips, Tricks, Pros/Cons & Advice

### Tips & Tricks

**Always write OCF members first, even empty:**
```cpp
class Foo {
public:
    Foo();
    Foo(const Foo& other);
    Foo& operator=(const Foo& other);
    ~Foo();
};
```
Fill in the body later. Never submit a class without all four.

**Use initializer lists in constructors — it's faster and the right style:**
```cpp
// Slow — default-constructs first, then assigns
Contact::Contact() {
    m_first_name = "";
    m_index = 0;
}

// Fast and correct — constructs directly with the right value
Contact::Contact() : m_first_name(""), m_index(0) {}
```

**`const` on every getter, always:**
```cpp
int getAmount() const { return _amount; }  // ✓
int getAmount()       { return _amount; }  // ✗ — you're lying to the compiler
```

**Check `this != &other` in `operator=`:**
```cpp
Account& Account::operator=(const Account& other) {
    if (this != &other) {  // protect against: a = a;
        _amount = other._amount;
    }
    return *this;
}
```

**When you get an `undefined reference` error, immediately check:**
1. Is there a static member without a definition in a `.cpp`?
2. Is there a method declared in the `.hpp` but never implemented?

### A Note on How to Think

The pattern you'll see throughout all CPP modules is:

> *The compiler enforces contracts at compile time so that bugs surface early rather than at runtime.*

`private` is a contract: "this data is managed only by me."
`const` on a method is a contract: "I will not change anything."
`const` on a parameter is a contract: "I will not modify what you passed me."

Every time you add one of these, you're not just following the norm — you're narrowing the space of possible bugs. A program that *can't* do the wrong thing is better than one that *shouldn't* do the wrong thing.

When you're unsure whether to add `const`, ask: *"is it physically possible for this function to need to modify this?"* If not — add `const`. Let the compiler be your safety net.

## Bonus — `private` and `public` at the Hardware Level

This is the question that separates people who *use* C++ from people who *understand* it.

**The short answer: at the hardware level, `private` and `public` do not exist.**

They are erased completely by the compiler before the program ever runs. The CPU has never heard of them.

---

### What an Object Actually Is in Memory

An object is just a **contiguous block of bytes** at some address. Private members, public members — they all live side by side in the same block. The CPU doesn't label them.

```
Contact object at address 0x7ffd3a20:

  Offset   Field              Access in C++
  ┌──────┬──────────────────┬───────────────────────┐
  │ +0   │ m_first_name     │ private               │
  │ +32  │ m_last_name      │ private               │
  │ +64  │ m_nickname       │ private               │
  │ +96  │ m_phone_number   │ private               │
  │ +128 │ m_darkest_secret │ private               │
  │ +160 │ m_index          │ private               │
  └──────┴──────────────────┴───────────────────────┘

From the CPU's perspective:
  READ  0x7ffd3a20 + 0    → just bytes, no label
  READ  0x7ffd3a20 + 160  → just bytes, no label
  WRITE 0x7ffd3a20 + 96   → just bytes, no label
```

---

### Where the Enforcement Actually Happens

`private` is enforced **at compile time**, not at runtime.

```
Your source code                  After compilation
────────────────                  ─────────────────

contact.m_first_name = "x";  ──► compiler REJECTS this line
                                  "error: m_first_name is private"
                                  → program never even builds

contact.setName("x");        ──► compiler ACCEPTS this line
                                  → generates: MOV [base + 0], "x"
                                  → pure memory write, no guard
```

```
Timeline:

  Compile time       Link time       Runtime
  ──────────────     ─────────────   ──────────────────────────────
  ┌────────────┐     ┌───────────┐   ┌──────────────────────────┐
  │ compiler   │     │ linker    │   │ CPU                      │
  │            │     │           │   │                          │
  │ checks     │     │ resolves  │   │ executes instructions    │
  │ private /  │     │ addresses │   │ reads/writes memory      │
  │ public     │     │           │   │                          │
  │ access     │     │           │   │ has NO idea what         │
  │ rules      │     │           │   │ "private" means          │
  └────────────┘     └───────────┘   └──────────────────────────┘
       ▲
       │
  access control
  lives and dies here
```

Once the compiler says "ok", `private` is gone. It produced a memory offset. The CPU just uses the offset.

---

### Proof: You Can Break `private` Without the CPU Caring

Because `private` is a compile-time rule, you can bypass it with tricks that the compiler can't see through:

```cpp
class Secret {
private:
    int password;
public:
    Secret() : password(1234) {}
};

Secret s;

// Method 1 — raw pointer cast
int* hack = reinterpret_cast<int*>(&s);
std::cout << *hack;  // prints 1234 — private means nothing here

// Method 2 — the infamous 42 trick (don't do this in real code)
#define private public
#include "Secret.hpp"   // now everything is "public"
```

This doesn't break any CPU rule — there are no CPU rules about access. It only bypasses the compiler's source-level check. Once compiled, there's nothing left to bypass.

---

### `this` Is Just a Pointer to the Base Address

When you call `contact.setName("Alice")`, the compiler rewrites it as:

```cpp
// What you write:
contact.setName("Alice");

// What the compiler actually generates (conceptually):
Contact::setName(&contact, "Alice");
//               ^^^^^^^
//               this pointer — address of the object's memory block
```

Inside `setName`, `m_first_name` becomes `*(this + 0)` — a raw memory offset from the object's base address. Public or private, the generated instruction is identical.

---

### Stack vs Heap — Where the Object Lives

Whether memory is on the stack or heap changes *where* the bytes are, not *what* access control means:

```
Stack allocation:              Heap allocation:
Contact c;                     Contact *c = new Contact();

Stack memory:                  Heap memory:
┌──────────────────────┐       ┌──────────────────────────────┐
│ main() frame         │       │ ...other heap data...        │
│  ┌────────────────┐  │       │ ┌────────────────────────┐   │
│  │ c              │  │       │ │ Contact object         │   │
│  │ m_first_name   │  │       │ │ m_first_name           │   │
│  │ m_last_name    │  │       │ │ m_last_name            │   │
│  │ ...            │  │       │ │ ...                    │   │
│  └────────────────┘  │       │ └────────────────────────┘   │
│ other locals...      │       │ ...other heap data...        │
└──────────────────────┘       └──────────────────────────────┘
      freed automatically            freed when you call delete c
      when scope ends
```

In both cases, the layout is the same block of bytes. `private` behaves identically on stack and heap.

---

### The Real Purpose of `private`

Now that you know `private` is a compile-time illusion, you can appreciate what it's actually for:

```
It is not protecting memory from the CPU.
It is not protecting memory from the OS.
It is not protecting memory from hackers.

It is protecting your design from yourself and your teammates.
```

When you mark `m_amount` as `private`:
- You make it impossible to accidentally write `account.m_amount = -9999` from outside the class
- You force all mutations through `makeDeposit()` and `makeWithdrawal()`, which contain your logic and validation
- Six months later, when you want to add logging to every deposit, you change *one function* instead of hunting down every direct assignment across the codebase

`private` is a **communication tool** and a **architectural contract**, enforced by the compiler as a favour to you. The hardware has nothing to do with it.

> Think of it like a "DO NOT ENTER" sign on a door. The sign doesn't physically stop you — the door isn't even locked. But it communicates intent, and a good team respects it. The compiler is the teammate that *always* respects it, and will yell at anyone who doesn't.

---

### Is `private` Comparable to a Mutex?

Good instinct — both are about controlled access to data. But they solve **completely different problems**.

```
                 private / public          mutex
                 ────────────────          ─────
  When?          compile time              runtime
  Against what?  wrong usage (logic)       concurrent access (threads)
  How?           compiler refuses to       thread is blocked / suspended
                 compile the code          until the lock is released
  Enforced by?   compiler                  OS / CPU atomic instructions
  Can bypass?    yes (pointer cast)        no — hardware-level
  Cost?          zero (erased at build)    real overhead (lock/unlock)
```

---

#### The Problem Each One Solves

**`private` solves a design problem:**

```
Thread A                         Thread A
────────────────────             ────────────────────
account.m_amount = -999; ✗      account.makeWithdrawal(999);
                                 → "refused" if amount insufficient ✓
compiler stops this              your logic handles this
at build time
```

**Mutex solves a time problem:**

```
Thread A                Thread B
────────────────────    ────────────────────
reads  _amount (42)
                        reads  _amount (42)
writes _amount (42-10)
                        writes _amount (42-5)  ← uses stale value!
                                               ← correct value should be 27
```

Without a mutex, two threads can read the same value simultaneously and both write back a result based on stale data. This is a **race condition** — it doesn't matter how well-designed your interface is, `private` can't help here.

---

#### Where They Actually Meet

Here's the connection: **`private` is the foundation you need before a mutex is even useful.**

If `m_amount` were `public`, any thread could write to it directly, bypassing your mutex entirely. The whole protection would be pointless.

The correct pattern combines both:

```cpp
class Account {
private:
    int         m_amount;  // private — no direct access from outside
    std::mutex  m_lock;    // private — only our methods can lock/unlock

public:
    void makeDeposit(int amount) {
        m_lock.lock();       // runtime: block other threads
        m_amount += amount;  // safe — only one thread at a time
        m_lock.unlock();
    }
};
```

```
Without private:               With private + mutex:

Thread A: account.m_amount++  Thread A: account.makeDeposit(1)
Thread B: account.m_amount++  Thread B: account.makeDeposit(1)
          ▲                             ▲
          both bypass the mutex         both go through the locked method
          race condition                serialized, correct
```

---

#### The Summary

| | `private` | `mutex` |
|---|---|---|
| Protects from | bad design / misuse | concurrent threads |
| Lives at | compile time | runtime |
| Hardware? | no | yes (atomic ops) |
| Performance cost | zero | real (microseconds) |
| Can be bypassed? | yes, with pointer tricks | no |
| You need it in CPP00? | yes | no (single-threaded) |

`private` says: *"you are not allowed to touch this."*
A mutex says: *"you can touch this, but wait your turn."*

They're complementary, not equivalent. In a multithreaded program, you need both: `private` to enforce the interface, mutex to serialize the access.

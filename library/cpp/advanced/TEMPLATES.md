# Templates — Complete Guide

Templates are C++'s answer to: "how do I write code that works for any type, without giving up type-safety?"

```cpp
template <typename T>
T max(T a, T b) { return a > b ? a : b; }

max(3, 5);                   // T = int         → 5
max(3.14, 2.71);             // T = double      → 3.14
max(std::string("a"), std::string("b"));    // T = std::string → "b"
```

One source; three real functions in the final binary, each generated by the compiler at the call site.

## 1. Why templates exist

Without templates, "write a function that works for any type" has three bad options in C:

```c
// 1. Duplicate for every type — maintenance hell
int    max_int(int a, int b);
double max_dbl(double a, double b);

// 2. Use void* and size_t — loses type safety, needs runtime dispatch
void max_generic(void* a, void* b, void* out, size_t sz, int (*cmp)(void*,void*));

// 3. Macro — no type checking, horror with expressions
#define MAX(a, b) ((a) > (b) ? (a) : (b))
```

Templates give you option 1's speed and safety with option 3's flexibility and *fully type-checked at compile time.* The compiler generates a bespoke function for each type you use — so `max<int>` is as fast as hand-written `max_int`.

## 2. Function templates

### Declaration

```cpp
template <typename T>
T max(T a, T b) {
    return a > b ? a : b;
}
```

- `template <typename T>` — "what follows is parameterized by a type called `T`."
- Inside the function, `T` stands in for whatever type the caller picks.

### Calling

Usually the compiler **deduces** `T` from the arguments — you don't need to spell it out:

```cpp
max(3, 5);           // int
max(3.14, 2.71);     // double
max<int>(3, 5);      // explicit — also legal, rarely needed
```

### Multiple template parameters

```cpp
template <typename T, typename U>
void iter(T* array, size_t len, U fn) {
    for (size_t i = 0; i < len; ++i)
        fn(array[i]);
}
```

`U` can be a function pointer, a functor (class with `operator()`), whatever. This is CPP07 ex01.

## 3. Class templates

A **whole class** parameterized by a type:

```cpp
template <typename T>
class Array {
    T*     _data;
    size_t _size;
public:
    Array() : _data(NULL), _size(0) {}
    Array(size_t n) : _data(new T[n]()), _size(n) {}
    ~Array() { delete[] _data; }

    T&       operator[](size_t i)       { return _data[i]; }
    const T& operator[](size_t i) const { return _data[i]; }
    size_t   size() const               { return _size; }
};

Array<int>           ints(10);
Array<std::string>   names(5);
Array<Dog*>          dogs(3);
```

`Array<int>` and `Array<std::string>` are *completely separate classes* — two full sets of methods compiled independently.

### Key syntactic thing

When you mention the class by name **outside its own definition**, you include the template parameters:

```cpp
template <typename T>
Array<T>::Array(size_t n) : _data(new T[n]()), _size(n) {}
//    ^^^
//    NOT just "Array"
```

## 4. Template arguments — three flavors

Template parameters can be:

### 1. Type parameters — the normal case

```cpp
template <typename T>    class Vec;
template <class T>       class Vec;    // synonym — see §5
```

### 2. Non-type parameters — a *value* known at compile time

Integers, pointers to globals, enums:

```cpp
template <typename T, size_t N>
class FixedArray {
    T _data[N];
public:
    size_t size() const { return N; }
};

FixedArray<int, 10> arr;        // N = 10 is baked into the type
FixedArray<int, 20> arr2;       // DIFFERENT type from arr
```

The value must be a compile-time constant. Passing a variable won't work.

### 3. Template template parameters — a *template* as a parameter

Rare; used by sophisticated container code:

```cpp
template <typename T, template <typename> class Container>
class Stack {
    Container<T> _data;
};

Stack<int, std::vector> s;      // Container = std::vector (as a template itself)
```

You won't touch these at 42. They're mentioned only so you recognize them in the wild.

## 5. `typename` vs `class`

In a template parameter list, `typename` and `class` are **synonyms**:

```cpp
template <typename T> void f(T);
template <class    T> void f(T);        // identical
```

Convention: use `typename` for "any type," `class` when you know it'll be a class type. Either is correct; `typename` reads more clearly when `T` might be `int` or `double`.

Elsewhere, `typename` has another job: disambiguating **dependent types** inside a template:

```cpp
template <typename T>
void f() {
    typename T::iterator it;    // "T::iterator IS a type" — tells the compiler
}
```

Without `typename`, the compiler can't tell if `T::iterator` is a type or a value (it doesn't know what `T` is yet). This is a C++98 quirk you'll bump into in STL-adjacent code.

## 6. Template argument deduction

The compiler works out `T` from the call:

```cpp
template <typename T> void f(T x);

f(42);             // T = int
f(3.14);           // T = double
f("hello");        // T = const char*
```

It matches the argument types to the parameter pattern. If the pattern is `T`, `T` becomes the argument type. If `T&` or `const T&`, the compiler strips the reference for deduction purposes.

### When deduction fails

- Return types are **not deducible** from the call — only from the arguments:
  ```cpp
  template <typename T> T zero();
  int n = zero<int>();            // must specify T explicitly
  ```
- Conflicting deductions:
  ```cpp
  template <typename T> void f(T a, T b);
  f(3, 3.14);                     // ERROR — T can't be both int and double
  ```

## 7. Specialization

Sometimes "one size fits all" doesn't. You want a custom implementation for a specific type.

### Full specialization

```cpp
template <typename T>
void print(T x) { std::cout << x << '\n'; }

// Specialization for bool
template <>
void print<bool>(bool x) { std::cout << (x ? "true" : "false") << '\n'; }

print(42);        // "42"
print(true);      // "true"  — specialized version
```

The `template <>` says "no new template parameters — this is a concrete version for `bool`."

### Partial specialization (classes only)

You can specialize part of the parameter list:

```cpp
template <typename T>          class Box;                 // primary
template <typename T>          class Box<T*> { /* ... */ };   // for any pointer type
template <typename T, size_t N> class Box<T[N]>;          // for arrays

Box<int>    b1;       // primary
Box<int*>   b2;       // pointer specialization
```

**Function templates cannot be partially specialized** — you overload them instead.

## 8. Why templates live in headers

When you instantiate `max<int>` in `main.cpp`, the compiler needs the **full source of `max`** right there to generate the int version. If `max`'s body was in `utils.cpp`, the compiler in `main.cpp` can't see it.

```
              NOT THIS:                        THIS:

utils.hpp:    template<typename T>            template<typename T>
              T max(T a, T b);                T max(T a, T b) { return ... }
utils.cpp:    template<typename T>
              T max(T a, T b) { return ... }  (nothing — header has it all)

main.cpp:     uses max<int>                    uses max<int>
              ↓                                ↓
              compiler can't see body          compiler sees body — generates int version
              → link error                     → works
```

**Conventions:**

1. Put everything in `.hpp` / `.h`. Simple.
2. Put declarations in `.hpp`, definitions in a separate `.tpp` (or `.ipp`) included at the bottom of the `.hpp`. Looks cleaner:
   ```cpp
   // Array.hpp
   template <typename T>
   class Array { /* decls */ };

   #include "Array.tpp"   // at the bottom
   ```

Both work. At 42, pick the one that the subject or your instincts prefer; many students use option 2 once they learn it.

## 9. Member function templates

You can put a template on a single method of a (non-template or template) class:

```cpp
class Logger {
public:
    template <typename T>
    void log(const T& value) {
        std::cout << "[log] " << value << '\n';
    }
};

Logger l;
l.log(42);
l.log("hello");
l.log(std::string("oui"));
```

Each call generates a fresh instantiation. The class itself isn't templated; only the method is.

In a templated class:

```cpp
template <typename T>
class Array {
    T* _data;
public:
    template <typename U>        // inner template
    U getAs(size_t i) const { return static_cast<U>(_data[i]); }
};

Array<int> a(5);
double d = a.getAs<double>(0);
```

## 10. Templates and inheritance

Templates can be base classes:

```cpp
template <typename T>
class Container {
public:
    virtual ~Container() {}
    virtual void add(const T&) = 0;
};

class IntList : public Container<int> {
public:
    void add(const int& v) { /* ... */ }
};
```

And you can templatize the derived class:

```cpp
template <typename T>
class MyList : public Container<T> {
public:
    void add(const T& v) { /* ... */ }
};
```

Watch out: **when you inherit from a dependent base, you must qualify base members explicitly.**

```cpp
template <typename T>
class Derived : public Base<T> {
public:
    void f() {
        // method();             // ERROR — compiler doesn't know it's there
        this->method();          // OK — delay lookup to instantiation time
        Base<T>::method();       // also OK
    }
};
```

This is C++98's "two-phase name lookup" rule. Rarely needed but surprising.

## 11. In CPP07 — Whatever & Array

CPP07 walks you through templates from simplest to most useful.

### ex00 — `whatever.hpp` — function templates

Three free function templates:
```cpp
template <typename T> void   swap(T& a, T& b);
template <typename T> T      min(const T& a, const T& b);
template <typename T> T      max(const T& a, const T& b);
```
These must work for any type that supports `<` / `>` and assignment. The `const T&` on `min` and `max` is important — avoids copying expensive types.

### ex01 — `iter.hpp` — higher-order function template

```cpp
template <typename T, typename F>
void iter(T* array, size_t len, F fn) {
    for (size_t i = 0; i < len; ++i)
        fn(array[i]);
}
```

`F` can be a function pointer, a functor, or — if you've moved to C++11 — a lambda. The point: templates aren't limited to type parameters that describe *data*; they also describe *operations*.

### ex02 — `Array.hpp` — class template

Build a generic container that:
- Default-constructs with `size() == 0`.
- Takes a `unsigned int n` to create an array of `n` default-constructed `T`s.
- Has a working copy constructor and `operator=` (deep-copy — OCF!).
- Has `operator[]` with **bounds checking** that throws `std::exception` (or a subclass).
- Provides `size()`.

This is where you stitch together: templates, OCF, exceptions, and the right use of `new T[n]()` (value-initialization, so ints are zero) vs `new T[n]` (default-init only).

## 12. Templates vs the STL mindset

The entire STL is built on templates. When you understand templates, the STL stops feeling magical:

- `std::vector<T>` is *literally* a class template you could have written.
- `std::sort(begin, end, cmp)` is a function template that takes any iterator and any comparator.
- Iterators work because templates never "see" the iterator's real type — they just call `++it`, `*it`, `!=`, and whatever the type provides, the compiler binds at instantiation.

This style is called **duck typing at compile time**: if `T` supports the operations the template uses, it works. No inheritance required. No interface to implement.

See [STL.md](STL.md) once you've built a class template yourself.

## 13. Rules of thumb

- **Start with the primary template. Specialize only when necessary.**
- **Use `const T&` for parameters you don't modify** — avoids expensive copies when `T` is heavy.
- **Default-initialize with `T()`** when you need a "zero" of unknown type. Works for all built-in types and any class with a default constructor.
- **Template members need to be visible at instantiation** — that's why they go in headers or `.tpp`.
- **Prefer function templates over `void*` + function pointers.** Every time.
- **If a non-template does the job, use a non-template.** Templates are slower to compile and produce bigger binaries.

## 14. Gotchas

- **Definitions in .cpp → linker errors.** If the compiler can't see the body at the call site, it can't instantiate. Move it to the header (or `.tpp`).
- **`typename T::something` is mandatory** when `something` might be a type inside a template parameter. Otherwise the compiler assumes it's a value.
- **Two-phase lookup**: names that depend on `T` aren't looked up until instantiation. `this->method()` or `Base<T>::method()` is how you tell the compiler "trust me, it'll be there."
- **Template error messages are famously ugly.** A small mistake can produce hundreds of lines. Read the *first* error — everything after is fallout.
- **Template argument deduction ignores conversions.** `max(3, 3.14)` fails because `T` can't be both `int` and `double`; explicitly `max<double>(3, 3.14)` works.
- **`new T[n]` vs `new T[n]()`** — parentheses matter. `new int[5]` gives uninitialized ints; `new int[5]()` zero-initializes. For class types with user-written constructors, they're equivalent. This bites in CPP07 ex02.
- **Full specialization of member function templates must be outside the class.** In-class full specialization is not legal in C++98.
- **Code bloat**: every distinct instantiation is a separate compiled function. `Array<int>` and `Array<long>` on a 64-bit system look identical but cost you twice the text.
